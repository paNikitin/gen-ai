## 1. Новый инструмент `compare_periods`

**Сигнатура** (`tools.py`):

```python
def compare_periods(
    metric: str,    # key_rate | fx_USD | fx_EUR | fx_CNY | cpi | unemployment
    period_a: str,    # YYYY-MM или YYYY-MM-DD
    period_b: str,
) -> dict
```

Внутри вызываются `get_fx_rate`, `get_key_rate`, `get_inflation`, `get_unemployment` — без дублирования логики. Схема описана в `schemas.py`.

**Примеры запросов, где агент вызывает инструмент:**

| Запрос | Вызов в trace |
|--------|---------------|
| «Во сколько раз вырос курс USD с января 2022 по апрель 2026?» | `compare_periods(fx_USD, 2022-01, 2026-04)` → ratio 0.96 |
| «Насколько изменилась инфляция г/г между мартом 2023 и мартом 2024?» | `compare_periods(cpi, 2023-03, 2024-03)` → delta +4.21 п.п. |
| «Как изменилась ключевая ставка с февраля 2022 по сегодня?» | `compare_periods(key_rate, 2022-02, 2026-06-16)` → delta −4 п.п. |

## 2. Лог шагов в JSONL

Каждый прогон `agent.py` и `eval.py` дописывает строки в `starter/trace.jsonl` (режим `"a"`).

Формат:

```json
{"run_id": "70d7ed21-...", "ts": "2026-06-15T21:58:43+00:00", "step": 1, "call": "compare_periods", "args": {...}, "obs": {...}}
{"run_id": "70d7ed21-...", "ts": "2026-06-15T21:58:46+00:00", "step": 2, "final": "..."}
```

`run_id` — uuid4, один на весь прогон. После ваших запусков в файле **51 строка** (seed-сценарии + agent + eval).

## 4. Диагностика трёх ошибок (из `trace.jsonl`)

### Ошибка A: битый JSON в аргументах

**Тип:** битый JSON в аргументах tool-call

**Трасса** (run_id `216edbc4`, seed-сценарий):

```json
{"step": 1, "call": "calculate", "args": {}, "obs": {"error": "битый json аргументов: Expecting value: line 1 column 16 (char 15)"}}
```

**Причина:** модель сгенерировала `{"expression": (1+2}` — невалидный JSON. `json.loads` в `_exec_one` падает.

**Что бы исправил:** retry с просьбой вернуть валидный JSON; примеры в схеме `calculate`.

---

### Ошибка B: галлюцинация инструмента

**Тип:** галлюцинация инструмента

**Трасса:**

```json
{"step": 1, "call": "get_exchange_rate", "args": {"currency": "USD"}, "obs": {"error": "неизвестный инструмент: get_exchange_rate"}}
```

**Причина:** модель выдумала `get_exchange_rate` вместо `get_fx_rate`. В `TOOLS_IMPL` такого имени нет.

**Что бы исправил:** в system prompt перечислить точные имена; в описании `get_fx_rate` указать «не используй get_exchange_rate».

---

### Ошибка C: инструмент вернул error — нет данных (реальный прогон eval)

**Тип:** инструмент упал / нет данных в источнике

**Трасса** (run_id `010333ea`, Q3 eval):

```json
{"step": 1, "call": "get_inflation", "args": {"year": 2026, "month": 5}, "obs": {"error": "нет данных ИПЦ на 2026-05"}}
{"step": 2, "call": "get_inflation", "args": {"year": 2026, "month": 4}, "obs": {"error": "нет данных ИПЦ на 2026-04"}}
{"step": 3, "call": "get_inflation", "args": {"year": 2026, "month": 3}, "obs": {"year": 2026, "month": 3, "cpi_yoy": 6.8, ...}}
```

**Причина:** CSV Росстата содержит ИПЦ только до марта 2026, а агент по умолчанию запрашивает «текущий» месяц (май/апрель). Инструмент корректно вернул `error`, агент переиграл на предыдущий месяц.

**Что бы исправил:** в промпте указать «если нет данных за текущий месяц — бери последний доступный»; в `get_inflation` добавить auto-fallback на последнюю строку CSV.

## 5. Вывод: самый хрупкий элемент

**Самое слабое место — согласованность tool-call между LLM и кодом:** имена инструментов, обязательные аргументы, валидность JSON. Это видно и в seed-сценариях, и в реальных прогонах (лишние попытки `get_inflation`).

Второе по хрупкости — **границы данных CSV vs «сегодня»**: агент не знает, до какого месяца есть Росстат, и тратит 2–3 шага на перебор.

`compare_periods` наоборот **упрощает** жизнь: Q5–Q7 решены за 1 вызов вместо пары `get_fx_rate` + `calculate`.
