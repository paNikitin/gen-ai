**Трек B** (прикладной).

---

## 1. Задача и данные

### Задача

Построить рабочий конвейер **Voice of Customer** для мобильного банка **NovaPay**: из сырых отзывов получить структурированную аналитику, проверяемую сводку, RAG-ответы по корпусу, агентные запросы с инструментами и мультиагентный обзор. Дополнительно — синтетическая мини-фокус-группа (5 персон).

### Данные

| Параметр | Значение |
|----------|----------|
| Источник | Синтетический корпус, написанный для финального проекта |
| Файл | `input/reviews.txt` |
| Объём | 40 отзывов, ~12 000 символов |
| Приложение | **NovaPay** (вымышленный банк) |
| Связь с семинарами | **Нет** — не МойБанк, не семинар 3 |

**Подготовка:** блоки `=== REVIEW N ===`, метаданные (автор, платформа, рейтинг 1–5, дата). Целостность: `python scripts/build_corpus.py`.

## 2. Система

### Техники курса (6)

| # | Техника | Модуль | Назначение |
|---|---------|--------|------------|
| 1 | Information Extraction + валидация | `schema.py`, `pipeline.py` | JSON-отзывы с issues, rating 1–5 |
| 2 | Map-Reduce + LLM-as-judge | `pipeline.py` | Сводка + проверка action items |
| 3 | Ghost-check цитат | `pipeline.py` | Подстрочное совпадение quote ⊂ корпус |
| 4 | Hybrid RAG (dense+BM25+RRF) | `rag.py` | Ответы с источниками |
| 5 | Агент с tools | `agent.py` | search_reviews, stats_by_platform, top_issues |
| 6 | Мультиагент (planner→workers→synth) | `orchestrator.py` | Сложные аналитические вопросы |
| + | Синтетические персоны | `personas.py` | 5 сегментов → `focus_group.json` |

### Схема пути данных

```
input/reviews.txt
    │
    ├─► pipeline.py ──► output/reviews.json, aspects.json, summary.json
    │                   judge_report.json, metrics.json, ghost_quotes.json, heatmap.png
    │
    ├─► rag.py ingest ──► chroma_db/ + output/bm25_cache.json
    │       └─► ask() / agent.search_reviews ──► ответ + sources
    │
    ├─► agent.py ──► tool calls (trace в памяти → eval_trace.jsonl)
    │
    ├─► orchestrator.py ──► plan → parallel agents → синтез
    │
    └─► personas.py ──► output/focus_group.json (синтетика, явно помечена)
```

### Структурированный вывод

- Все LLM-вызовы через `response_model` + `max_retries` (Pydantic).
- Бизнес-инвариант: `@field_validator` — дата отзыва не в будущем; рейтинг 1–5.

## 4. Где пайплайн ошибался

### Пример 1 — RAG: неполный retrieval (кейс #5)

| | |
|---|---|
| **Вход** | «Сколько ждали поддержку…»; gold: `review_1` (сутки) и `review_11` (**4 дня**) |
| **Retrieved top-5** | review_5, review_1, review_3, review_20, review_7 — **без review_11** |
| **Искажение** | Ответ: «ждали сутки», хотя в корпусе есть более тяжёлый кейс (4 дня). Hit-rate = 0.5 |
| **Причина** | top-k=5 + RRF: review_11 семантически дальше от формулировки запроса |

### Пример 2 — RAG: сглаживание полярности (кейс #4)

| | |
|---|---|
| **Вход** | Запрос про **устаревший** дизайн (review_4, review_37) |
| **Ответ модели** | «…устаревший… **в то время как другие находят современным**» |
| **Искажение** | Вопрос односторонний, ответ балансирует позитивные отзывы (review_7, review_23), размывая фокус |
| **Причина** | LLM стремится к нейтральному summary при смешанном контексте |

### Пример 3 — Агент: лишние инструменты (кейс #12)

| | |
|---|---|
| **Вход** | «Сравни отзывы про поддержку»; ожидался `search_reviews` |
| **Путь** | Вызваны **3** tools: search_reviews + stats_by_platform + top_issues |
| **Искажение** | Ответ сводит поддержку к «трём лагерям» и severity-скорам — аналитически богато, но narrative сжат и обобщён |
| **Причина** | Модель переиспользует tools «на всякий случай»; нет штрафа за избыточные вызовы в eval |

**Ghost-check:** 0 ложных цитат из 57 — обобщения в аспектах не прошли бы проверку, если бы quote не совпал с подстрокой корпуса.

## 5. Выводы

### Ответ на задачу

Конвейер NovaPay VoC **решает постановку**: из 40 отзывов получаем структурированную аналитику, judge-проверенную сводку (8 action items), RAG и агентные ответы. Топ pain points: **надёжность** (severity 42), дизайн (18), реклама (17), поддержка (16).

### Границы

- Синтетические отзывы и персоны — не реальные пользователи.
- Одна LLM, CLI без UI.
- Ghost-check — подстрочное совпадение первых 30 символов (не семантическая проверка).
- RAG top-k=5 ограничивает полноту на редких кейсах.

### Что оставить в прод / что переделать

| В прод | Переделать |
|--------|------------|
| IE + Pydantic-валидация | Увеличить top-k / reranker для RAG |
| Ghost-check перед публикацией цитат | Штраф агенту за лишние tool calls |
| Hybrid RAG для ad-hoc вопросов | Отдельный числовой fact-check для агента |
| Judge как gate для action items | Калибровка judge (сейчас 1.0 — возможен оптимизм) |
